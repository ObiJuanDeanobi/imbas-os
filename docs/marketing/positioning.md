# Imbas OS positioning

This is a working go-to-market positioning note for Imbas OS. It is not a public launch page yet.

## Category

Imbas OS is a **local-first agent workbench**.

It is not just:

- an AI memory app;
- a notes app;
- an artifact viewer;
- a RAG dashboard;
- an agent runner.

It is the operating layer where AI agent work becomes durable, reviewable, searchable, and portable.

## One-line positioning

> Imbas OS is a local-first home base for your AI agents — capturing artifacts, memory, wiki updates, run history, context packs, and human approvals in one private workspace.

## Short positioning

AI agents are useful, but their work is scattered across chat history, terminals, repos, downloads, and half-remembered prompts. Imbas OS gives agents a durable local workspace: generated artifacts, living wiki pages, context events, run history, snapshots, and approval trails all live together.

The promise is simple:

> Your AI work should not evaporate when the chat ends.

## Primary audience

### 1. Agent builders and AI power users

People already using tools like OpenClaw, Claude Code, Codex, Cursor, local scripts, Obsidian, and custom automations.

Pain:

- agent work is scattered;
- context is lost between runs;
- useful artifacts vanish into chat/downloads;
- no durable audit trail.

Message:

> Give your agents one durable home base.

### 2. Solo devs and indie builders

People building serious projects with AI help.

Pain:

- AI can move fast, but project memory gets chaotic;
- decisions and generated assets are hard to reuse;
- context handoff between coding sessions is fragile.

Message:

> Turn AI-assisted work into durable project memory.

### 3. Evidence-heavy knowledge workers

Compliance, security, research, analysis, and other work where provenance matters.

Pain:

- need source trails, review steps, and defensible outputs;
- AI output needs human approval and evidence packaging;
- auditability matters.

Message:

> AI-assisted work with receipts.

### 4. Small teams experimenting with agents

Later audience, after the local core and cloud/workspace product mature.

Pain:

- multiple agents/tools with no shared state;
- review and approvals are ad hoc;
- no common run ledger.

Message:

> A shared control plane for agent work.

## Messaging pillars

### Local-first and private

Your agent memory, artifacts, wiki, and run history live with you first.

### Agent work becomes durable

Imbas OS captures artifacts, context events, proposals, snapshots, and Runledger evidence.

### Human review is built in

Lorekeeper proposals, approve/apply flows, snapshots, restore, and audit trails make agent changes inspectable.

### Bring your agents

OpenClaw first; Hermes, Codex, Claude Code, and other connectors later.

### More than memory

Memory alone is not enough. Imbas OS combines memory, artifacts, wiki, run history, approvals, context packs, and mobile capture.

## Do not lead with

- internal subsystem names before the problem is clear;
- “autonomous AI OS” hype;
- enterprise procurement language;
- generic “second brain” positioning;
- “Obsidian replacement.”

Subsystem names are useful after the user understands the product:

- Memsocket = memory/context;
- Artifact Vault = generated artifacts;
- Lorekeeper = living wiki/review;
- Conduit = connectors/API;
- Runledger = run/audit trail;
- Sanctum = trust/redaction/approvals.

## Best short lines

- A home base for your AI agents.
- Make AI work durable.
- Local-first memory and control for agentic work.
- Your agents remember here.
- From chat chaos to durable work.
- AI-assisted work with receipts.

## Recommended first public narrative

> I built a local-first command center where OpenClaw can save artifacts, write reviewed wiki updates, keep a run ledger, build context packs, and pair with my phone.

That story is concrete, demoable, and honest.
